34.5M 参数赢过 235B 大模型，三档全测完：PP-OCRv6 真强还是假强

作者：卡兹克 · 首发：2026-06-15 · 三档补测：2026-06-16 · 「数字生命卡兹克」

前段时间我写过一篇 OCR 选型实测，标题大概是：别瞎花钱了，我实测了 18 个文档、6 类场景，教你选对 OCR 模型。

那篇文章里，我当时的判断很简单：OCR 不是一个模型打天下。规整文本、手写、表格、双栏论文、公式文档，背后其实是几种完全不同的能力。你要的是逐字还原，还是文档理解；你能不能联网，数据能不能出本地；这些问题比”哪个模型分数最高”更重要。

结果那篇刚写完没多久，PaddleOCR 又丢了一个新东西出来：PP-OCRv6。

我一开始其实没太在意，OCR 更新嘛，正常，每年都有。直到我看到两行字，第一行是「Tiny 版本只有 1.5MB，可以直接跑在浏览器里」，第二行是「Medium 档只有 34.5M 参数，OCR 专项任务上超过了 Qwen3-VL-235B」。

我寻思了一下没寻思明白，34.5M 对 235B，6906 个汉字字符 vs 万亿 token 预训练，为什么专用小模型能在这件事上赢？

然后就是一个问题跟着另一个问题。OCR 的未来，真的是越来越大的 VLM 吗？还是说这条轻量专用模型的路，反而有自己的护城河？

这篇文章是上一篇的续集，也算是我对这个问题的亲手测验。我把 PP-OCRv6 Tiny、Small、Medium 三档都本地跑了一遍，沿用同一把 OmniDocBench 的尺子，加了四张真实世界难图，顺便做了个本地 OCR 工作台。说说我看到的结果，以及我自己对这件事的判断。

· · ·

稍微说一下背景，免得有朋友没接触过这个领域。

OCR，全称 Optical Character Recognition，其实就是"把图片里的字变成可编辑文本"这件事。听起来简单，但做好很难，字体、排版、手写、表格、双栏、数学公式，每一个都是单独的挑战。

PaddleOCR 是百度开源的 OCR 系统，在国内工程圈用得很广，做文档处理的基本都碰过它。PP-OCRv6 是这套系统的最新版，六月十一号刚发的。

现在主流的 OCR 有两条路，一条是传统的检测+识别流水线，先框出文字区域，再逐框读出文字，这条路轻量，可以在端侧跑。另一条是用多模态大模型直接看图说字，像 GPT-4o、Qwen3-VL 那样，精度高但很重，必须调云端 API。

PP-OCRv6 走的是第一条路。但这次我想亲手戳一戳的，不只是“它是不是又刷榜了”，而是它到底怎么用这么小的模型，去和一堆大模型抢 OCR 这个活。

如果只看发布话术，很容易写成“34.5M 参数干翻 235B 大模型”。这句话很爽，但我觉得不够准确。OCR 不是一个单点能力，它至少拆成三件事：能不能找到字，能不能逐字读准，能不能理解版式顺序。

PP-OCRv6 真正厉害的是前两件事。它给检测和识别用了同一套 LCNetV4 骨干：检测时正常压缩图片去找文字位置，识别时只压缩高度、尽量保留宽度，把一行字当成序列来读。一个底座服务两个任务，Tiny、Small、Medium 三档都能复用。

我理解的技术重点：
1）LCNetV4 骨干，检测和识别共用一套，三档（Tiny/Small/Medium）通用架构，区别在宽度和深度；
2）检测侧把感受野从 3×3 扩到 7×7，参数反而更少，小字、密集文字检测效果明显提升；
3）识别侧加了轻量注意力模块，处理字符之间的上下文；字典扩展约 200 个带重音字符，一个模型覆盖中、英、日和 46 种拉丁语系共 50 种语言；
4）三档官方检测/识别分（PP-OCRv6 论文）：Medium 86.2 / 83.2，Small 84.1 / 81.3，Tiny 80.6 / 73.5；整体超越 Qwen3-VL-235B（检测 38.3，识别 74.9）、GPT-5.5 等大模型。
5）路线不是”越大越好”，而是把 OCR 做成一套可按场景部署的梯队：Tiny 浏览器/嵌入式，Small 移动端/本地应用，Medium 服务器/高精度本地。

所以如果标题再狠一点，也可以写成：PP-OCRv6，34.5M 参数干翻 235B 大 VLM 模型。但我更愿意把它翻译成人话：它不是靠堆参数赢，而是把 OCR 这件事拆得足够细，检测、识别、字符集、部署规格都重新做了一轮。

检测和识别两个 OCR 专项任务上，PP-OCRv6 都是在和上一代以及主流 VLM 拉开差距。
Tiny / Small / Medium 三档的意义不只是大小不同，而是覆盖浏览器、移动端、本地应用和服务器部署。
· · ·

为了和上一篇选型实测接上，我这次还是沿用同一把尺子：OmniDocBench。它专门用来评测文档 OCR，18 张文档图片，涵盖中文电子书、英文学术论文、报纸、教材、PPT 幻灯片、手写笔记六种场景，每张都有人工标注的 ground truth，用 text_block 编辑距离（quick_match）打分，分越低越好。

这次跑了六个选手，本地派三个档，云端参照两个，

PP-OCRv6 Tiny，1.5MB，onnxruntime-web 跑在 Chrome 浏览器里，WebGPU 加速，图片全程不出本地。

PP-OCRv6 Small，7.7MB，本地 ONNX Runtime + CoreML，Apple Silicon 加速，补测新增。

PP-OCRv6 Medium，34.5MB，本地 ONNX Runtime + CoreML，三档里精度最高，补测新增。

Apple Vision，macOS 原生 OCR，accurate 档，完全离线，作为本地基线对比。

GLM-OCR，智谱云端 OCR API，参照系。

PaddleOCR-VL-1.5，百度多模态大模型云端 API，参照系。

前四个不联网，后两个调云端。这里我把三档 PP-OCRv6 都跑了一遍，是因为我想看一个问题：同样是检测+识别的流水线，模型大一点，在 OCR 这件事上到底能带来多少提升。

先放一张浏览器跑起来的截图，因为我自己刚看到那一行"后端：webgpu"的时候，确实愣了一秒。

Chrome 浏览器内加载 PP-OCRv6_tiny ONNX 模型，后端自动切换为 WebGPU
18 张图推理完成，每张显示检测时间、识别时间、框数和识别文本

1.5MB 的模型文件，用的是 onnxruntime-web 这个 JS 库，页面里就一个 HTML 文件，没有后端，没有服务器，打开浏览器就跑，图片全程在本地处理。这玩意确实能做到。

不过单页 demo 只能证明“能跑”。我后来又往前推了一步：既然它能在浏览器里跑，那能不能做成一个真正可用的小工作台？中文能不能稳定识别？表格能不能看？能不能下载？遇到真实世界照片，Tiny 不行的时候，换 Small 或 Medium 会不会明显好一点？

这里我觉得产品体验页里的案例设计得挺聪明。它没有只放干净的扫描件，而是直接把 OCR 最容易翻车的场景摆出来：随手拍的卷曲试卷、多栏排版报道、竖向阅读文字、手写数学笔记、活动海报、高数推导、化学教材公式、繁体字杂志、表格里的复杂公式。

这些案例基本覆盖了非平面、特殊排版、手写、公式、表格、多语言几类核心难点。

这张图给我的提醒是：OCR 早就不是“拍一页 A4 纸读出来”这么简单了。真正落地时，用户会拿歪的、皱的、竖排的、手写的、带表格公式的、图文混排的内容来测你。能不能识别是一层，能不能保留阅读顺序、表格结构、公式关系，又是另一层。

于是我补了一版本地 OCR Studio。后端用 FastAPI + ONNX Runtime，Apple Silicon 上走 CoreML；前端保留拖拽上传、历史记录、可视化框选、行明细。表格这块没有上 PP-Structure，而是先用文字框的几何位置做行列聚类，识别后可以直接在页面里看表格，也可以导出 CSV、Markdown、HTML 和 Excel。

这次我把 Tiny、Small、Medium 三档都放到了本地。Tiny 是 1.5M，Small 是 7.7M，Medium 是 34.5M；ONNX 文件落到磁盘后会更大，但规格对应的是这三档官方模型。工作台里现在可以直接切换模型再上传同一张图测。

我又加了四张「文档不友好」的真实世界图片，一张斜拍名片、一张点阵字体、一张轮胎侧壁压印、一张电梯数码屏，透视、低对比、发光屏、特殊字体，都是 OCR 在真实场景里常见的边界条件。

名片和轮胎这两张的对比最直观，名片场景里多模态模型倾向补全语义，专用 OCR 更偏逐字抄写；轮胎压印场景里低对比浮雕字把大部分模型都逼到了边界。

名片场景，多模态模型容易补全语义，专用 OCR 更偏逐字抄写。
轮胎压印场景，低对比、弧面和浮雕文字会把大部分模型逼到边界。

五个选手，四张图，按场景说。

名片透视，斜拍加彩色底，透视角度让字体变形是最大的难点。Medium 最完整，DESIGN SOUL、JONATHAN DOE、Creative Designer、电话和 url 基本全读出来，Small 差不多。Apple Vision 主体读到但 DESIGN/Designer 有误读，PaddleOCR-VL 以文档解析为主，名片这类照片不是它的主场。

点阵字体，字形断裂对字符集要求高，反而是 Small 最稳，读出了 RETRO PRINT CHARN，Medium 稍差（RFTRO PRTNT CHARN）。Apple Vision 读出 RETRO PRINT CHARM 最接近正确，PaddleOCR-VL 把主标题当图片块跳过了。

轮胎压印，低对比浮雕字是边界场景的代表，肉眼都不容易看清的那种。Medium 整体最稳，读出 TREADWEAR 220、PLACARD IN VEHICLE、TO SEAT BEADS、ME AXLE，Apple Vision 关键数字 220 读到，TREADWEAR 形态接近，PaddleOCR-VL 轮胎这类图不是它的主场。

电梯数码屏，七段数码字体加反光金属底，对所有模型都有挑战。三档 PP-OCRv6 都读出了 BVY 标签和品牌网址，数字部分 Medium 最稳，ELEVATOR 有拼写误读，Apple Vision 类似，PaddleOCR-VL 把顶部屏幕当图片块跳过了。

速度参考，OmniDocBench 18 张均值，Tiny 大约 3-15 秒一张，Small 4-25 秒，Medium 10-52 秒，Apple Silicon CoreML 加速，Apple Vision 0.16-0.54 秒，PaddleOCR-VL 走云端大约 6.7-15.9 秒。

我看到 Apple Vision 那份结果，就有点无语，这还是 Apple 的亲儿子框架、accurate 档。手写中文确实难，但 PP-OCRv6 在这张上的编辑距离是 Tiny=0.283、Small=0.245、Medium=0.228，Apple Vision 是 0.570，差了整整一倍多。

我觉得这个数据本身就挺有说服力的。手写场景，参数越大越管用，这条路值得继续走。

当然，不是所有场景 PP-OCRv6 都占优，必须说失分的地方。

最大的坑是双栏排版。中文学术书籍、英文期刊论文，这类图片 PP-OCRv6 和 Apple Vision 都垮得很惨，编辑距离高达 0.8 到 0.9。

测试样本，中文电子书，典型双栏排版

为什么？原因其实挺直接的。检测+识别的流水线，检测阶段只知道"这里有一块文字，坐标是 xxx"，它不理解"左边这一列要先读完，再读右边那一列"这种版式逻辑。所以识别结果是按照文字框从上到下的空间顺序排列，把左右两列文字混编在一起了。

来看一下实际输出，就知道有多乱：

那发布会说的那些话，是真的吗？

这次我把 Medium 也跑了一遍，可以直接说了。

浏览器可以跑，真的，我亲测了。WebGPU 加速，单页大约 1.8 秒（平均 102 个文字框串行识别）。

三档都能在本地 ONNX Runtime + CoreML 上跑，Medium 每张大约 20-25 秒（Apple M1 Pro 上），慢是慢，但数据不出本地，CoreML 加速，不需要 API key，不需要联网。

1.5MB 模型手写识别打赢 Apple Vision，真的，数据在上面，差了将近一倍。而且这个差距随着模型变大还在扩大，手写笔记①，Medium=0.228、Small=0.245、Tiny=0.283、Apple Vision=0.570。Medium 比 Apple Vision 好了 2.5 倍。

双栏版式搞不定，这也是真的。检测+识别的架构本来就不处理阅读顺序，三档 PP-OCRv6 在中文双栏书籍①这张上全都是 0.897，和 Apple Vision 一样惨。这是结构性限制，不是 Medium 比 Tiny 大就能解决的问题。

· · ·

这里有件事我想多说几句，因为它是整个「为什么 34.5M 能赢 235B」这个问题的核心。

大模型做 OCR，有个很有趣的缺陷，幻觉。

不是那种「胡说八道」的幻觉，是更隐蔽的，它会把图片里写错的字「纠正」过来。比如手写笔记里的错别字，VLM 看到后，不是忠实地抄下来，而是把它读成正确的写法。这从「理解语义」的角度看是聪明的，但从「逐字还原」的角度看，这是一个 bug，不是 feature。

PP-OCRv6 技术报告里有一个数据，PP-OCRv6 精确匹配率 93.2%，Qwen3-VL-235B 是 80.6%。这个精确匹配率衡量的是「识别出的文字是否和图片里的文字完全一致」，不允许任何「聪明的修正」。

差了将近 13 个百分点。

所以如果你需要的是发票、合同、表单、票据这类对逐字准确性要求极高的场景，大模型的「补全语义」能力反而会变成风险。反倒是专用轻量模型，读到什么就是什么，没有脑补，更可靠。

这是我认为 PP-OCRv6 能在 OCR 专项任务上赢过大模型的根本原因，不只是「参数少效率高」，而是任务范式不一样，OCR 要的是抄写员，不是翻译家。

· · ·

说说我自己的感受吧。

三档跑完，我对 PP-OCRv6 的判断反而比一开始更清楚了。

Tiny 适合的场景，想在浏览器或者嵌入式端跑，1.5MB 的体积没得说，规整文档和手写都能用，跑完不出本地。Small 是一个甜点档，7.7MB，速度比 Tiny 稍慢，精度比 Tiny 稍好，尤其是手写场景的提升可以感知。Medium 是有意义的，在手写、彩色教材这类场景上比 Tiny 提升明显，但速度也更慢，适合对精度有要求、但又不想调 API 的本地部署场景。

三档有一件事是共通的，数据不出本地，不需要 API key，不需要后端，不需要服务器，一个本地工作台就能跑起来。识别完还能看表格、下 CSV/Excel、留历史记录。这对很多场景来说是刚需，比如要处理敏感文档的公司内网，比如离线环境，比如不想为每次 OCR 付 API 费用的个人开发者。

所以回到那个问题，OCR 的未来是不是越来越大的 VLM？

我的判断是，两条路各有各的场景，并不是一条路注定会吃掉另一条。大模型做 OCR，擅长的是「理解文档」，能处理双栏、公式、表格结构；轻量专用模型做 OCR，擅长的是「抄写文字」，逐字准确，无幻觉，能在端侧跑。当你需要的是逐字还原，不是文档理解，专用模型反而更靠谱。

所以别瞎花钱。如果你的文档是截图、票据、表单、手写笔记，PP-OCRv6 三档随便一个都够用，而且可能是更便宜、更可控、隐私更好的选择。如果你需要处理双栏论文、复杂版式报告，那现阶段还是调 PaddleOCR-VL 或者 GLM-OCR 更合适。不是 PP-OCRv6 不行，而是任务已经从”识别文字”变成了”理解文档”，是两件不同的事。

就这样，说清楚了。
